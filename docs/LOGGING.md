# Логирование

Этот документ описывает стандартный подход к логированию в системе, нацеленный на обеспечение наблюдаемости, упрощение отладки и анализа поведения компонентов. Ключевой принцип здесь простой: логи должны быть структурированными картами.
Имплементация шины событий доступна в [GitHub-репозитории](https://github.com/algebrain/lcmm-event-bus).

## 1. Стандартный Интерфейс Логгера

Логгер в системе представляет собой функцию, которая принимает два аргумента:
1.  `level` — уровень логирования (ключевое слово, например, `:info`, `:warn`, `:error`, `:debug`, `:trace`).
2.  `data` — **карта** с данными о событии, предоставляющая контекст.

### Пример реализации логгера
Вы можете реализовать логгер, который обертывает любую библиотеку логирования, например `clojure.tools.logging`:

```clojure
(require '[clojure.tools.logging :as log])

(defn make-app-logger []
  (fn [level data]
    (case level
      :info  (log/info data)
      :warn  (log/warn data)
      :error (log/error data)
      :debug (log/debug data)
      :trace (log/trace data)
      (log/info data))))) ; Дефолтный уровень
```

## 2. Интеграция Логгера в Компоненты

Каждый компонент или модуль системы, требующий логирования, должен принимать экземпляр этой функции-логгера в качестве зависимости.

Например:
*   **`event-bus`**: При создании экземпляра шины через `bus/make-bus` функция-логгер передается через опцию `:logger`.
*   **Модули**: Функция `init!` каждого модуля будет принимать логгер как одну из своих зависимостей, что позволит модулю логировать свои внутренние события.

## 3. Структура Лог-Сообщений

Все сообщения, передаваемые логгеру в аргументе `data`, **должны быть картами**. Это обеспечивает структурированность логов, упрощает их поиск, фильтрацию и анализ в системах агрегации логов.

Рекомендуемые поля для всех лог-сообщений:
*   `:component` (обязательно): Ключевое слово, идентифицирующее компонент, который сгенерировал лог (например, `:event-bus`, `:my-module`, `:user-service`).
*   `:event` (обязательно): Ключевое слово, кратко описывающее тип произошедшего события (например, `:event-published`, `:user-created`, `:http-request-received`).
*   Дополнительные поля: Любые другие данные, которые помогают понять контекст события (например, `correlation-id`, `user-id`, `path`, `exception`, `payload`).

## 4. Как связывать одну цепочку действий в логах

В LCMM логи должны помогать собрать в одну историю все, что произошло в рамках одного запроса или одной цепочки событий.

Обычно в эту цепочку входят:

1. HTTP-запрос;
2. публикация события в `event-bus`;
3. обработка производных событий;
4. аудит;
5. действия защитного слоя.

Практическое правило:

1. `correlation-id` связывает всю цепочку целиком;
2. `request-id` относится только к конкретному HTTP-запросу;
3. если HTTP-обработчик публикует событие в шину, тот же `correlation-id` должен быть виден и в логах следующих обработчиков.

Минимальный набор полей для таких записей:

1. `:component`
2. `:event`
3. `:correlation-id`
4. `:request-id`, если запись относится к HTTP-слою
5. `:path` или `:event-type`, если это помогает понять, где именно мы находимся в цепочке

Небольшой пример:

```clojure
(logger :info {:component ::http
               :event :http/request-finished
               :path "/bookings/actions/create"
               :status 200
               :correlation-id "c-123"
               :request-id "r-456"})

(logger :info {:component ::notify
               :event :notification-created
               :booking-id "b-1"
               :correlation-id "c-123"})
```

Смысл здесь простой: по `correlation-id` вы должны без труда найти в логах и входящий запрос, и то, какие события он породил дальше.

## 5. Где и когда скрывать чувствительные данные

Секреты нужно скрывать до того, как запись уйдет в систему логирования.
Нельзя рассчитывать, что это безопасно сделает кто-то дальше по цепочке.

Минимум ключей, которые нужно скрывать обязательно:

1. `password`
2. `token`
3. `secret`
4. `api-key`
5. `authorization`
6. `cookie`

Практическое правило:

1. скрывайте чувствительные поля рекурсивно, включая вложенные map;
2. исключения можно логировать, но без утечки секретных данных из payload;
3. если сомневаетесь, лучше логировать краткое описание, а не весь сырой запрос.

Минимальный пример:

```clojure
(logger :warn {:component ::security
               :event :security/guard-event
               :authorization "Bearer secret-token"
               :payload {:password "qwerty"}})
;; в sink должно попасть:
;; {:authorization "***"
;;  :payload {:password "***"} ...}
```

Здесь важна граница ответственности: защита логов от утечки секретов должна происходить в вашем коде, а не где-то потом.

## 6. Когда писать в логи, а когда в аудит

Логи и аудит решают разные задачи, и смешивать их не нужно.

Логи нужны для:

1. разборов сбоев и странного поведения;
2. трассировки запроса;
3. ошибок и деградаций;
4. событий защитного слоя.

Аудит нужен для:

1. фиксации значимых бизнес- или системных действий;
2. восстановления того, что произошло по доменным событиям;
3. более устойчивой истории действий, чем обычные рабочие логи.

Практические правила:

1. не каждое лог-событие должно попадать в audit;
2. audit не заменяет request/error/security logs;
3. лог и audit могут ссылаться на один `correlation-id`, но хранить разный объем данных.

Короткая опорная матрица:

1. HTTP request started/finished -> logs
2. guard degraded / rate-limited / banned -> logs
3. `booking/created` -> logs и, если это значимо для приложения, audit
4. admin unban -> logs; audit добавляется, если это операционно значимое действие

То есть вопрос обычно не в том, что "важнее", а в том, какую задачу решает запись: помогает разбирать работу системы сейчас или фиксирует значимое действие как часть истории.

## 7. Что стоит логировать на уровне приложения

Для небольшого и среднего приложения полезно иметь хотя бы такой минимум:

1. `:app/starting`
2. `:app/started`
3. `:app/start-failed`
4. `:http/request-started`
5. `:http/request-finished`
6. `:http/request-failed`
7. `:security/guard-event`

Это не жесткий стандарт имен, а отправная точка, которой обычно достаточно для рабочего приложения.

## 8. Короткий пример всей цепочки

Ниже короткий пример одной полной цепочки:

```clojure
(defn create-booking-handler [bus logger request]
  (logger :info {:component ::http
                 :event :http/request-started
                 :path (:uri request)
                 :correlation-id (:lcmm/correlation-id request)
                 :request-id (:lcmm/request-id request)})
  (bus/publish bus
               :booking/create-requested
               {:slot-id "slot-09-00" :user-id "u-alice"}
               (http/->bus-publish-opts request {:module :booking}))
  {:status 200})

(defn on-booking-created [bus logger envelope]
  (logger :info {:component ::notify
                 :event :notification-created
                 :booking-id (-> envelope :payload :booking-id)
                 :correlation-id (:correlation-id envelope)})
  (bus/publish bus
               :notify/booking-created
               {:booking-id (-> envelope :payload :booking-id)}
               {:parent-envelope envelope
                :module :notify}))
```

Что показывает этот пример:

1. лог о запросе и лог о следующем обработчике принадлежат одной цепочке через `correlation-id`;
2. audit может отдельно сохранить `booking/create-requested` или `notify/booking-created`;
3. logs и audit связаны, но не являются одной и той же системой.
